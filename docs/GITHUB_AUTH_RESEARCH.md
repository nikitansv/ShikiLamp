# GitHub research: authentication patterns for Lampa and Shikimori

Checked: 2026-07-13

## Repositories found

### `ARManakhov/shikimori_plugin`

URL: https://github.com/ARManakhov/shikimori_plugin

A real Lampa plugin with Shikimori OAuth UI. It provides:

- QR code pointing directly to Shikimori `/oauth/authorize`;
- out-of-band redirect URI `urn:ietf:wg:oauth:2.0:oob`;
- manual authorization-code input in Lampa;
- direct browser exchange at `/oauth/token`;
- access/refresh token storage in `Lampa.Storage`;
- automatic refresh based on `created_at + expires_in`;
- `whoami` after authorization.

Security assessment:

- The repository embeds a default OAuth client ID and client secret in public JavaScript. Values are intentionally omitted here.
- This makes its client secret public and unsuitable as a security model for ShikiLamp.
- The QR is generated by a third-party service (`barcode.orcascan.com`) and sends the full authorization URL to that third party.
- Authorization still requires the user to copy an authorization code from the phone into Lampa. It is easier than manually obtaining a token, but not a complete TV pairing flow.
- No OAuth `state` is present in the generated authorization URL.
- Tokens are stored locally, which is acceptable for a single-device client if logs/exports remain redacted.

Reusable ideas:

- Lampa settings UI structure;
- QR/on-screen instructions;
- local token expiry calculation;
- refresh-before-request behavior;
- `whoami` verification.

Do not reuse:

- embedded secret;
- missing `state`;
- third-party QR endpoint for sensitive authorization URLs;
- OOB/manual-code UX as primary onboarding.

License: GPL-2.0. Copying code would impose GPL obligations. Reimplement the pattern independently.

### `boiler4/lampa-plex-source`

URL: https://github.com/boiler4/lampa-plex-source

A maintained Lampa plugin with a good TV login UX using Plex's native PIN API:

```text
POST https://plex.tv/api/v2/pins
GET  https://plex.tv/api/v2/pins/:id
https://plex.tv/link
```

Behavior:

- obtains PIN ID and short code;
- shows code/QR/link in a TV overlay;
- polls every 2 seconds;
- expires after 2 minutes;
- stores token after authorization;
- supports cancel/back;
- keeps manual token input as fallback.

This is the closest Lampa UX reference for ShikiLamp.

Important limitation: Plex exposes a native PIN/device-like API. Shikimori does not. We can reuse the Lampa overlay/polling UX, but must implement equivalent session endpoints in our OAuth relay.

Security observations:

- PIN flow itself does not require embedding a client secret;
- QR uses third-party `api.qrserver.com`; for ShikiLamp prefer local QR generation or relay-hosted SVG/PNG to avoid leaking verification URLs;
- polling should use an unguessable session secret and POST body, not only a numeric ID.

License: MIT. Patterns/code can be reused with attribution and license compliance, but minimal independent implementation is still preferable.

### `Noilty/oauth2-shikimori`

URL: https://github.com/Noilty/oauth2-shikimori

PHP provider for `league/oauth2-client`. It confirms standard server-side OAuth provider integration:

- authorization endpoint;
- token endpoint;
- `whoami` resource-owner endpoint;
- standard authorization parameters/state behavior inherited from the OAuth client library.

Limitations:

- archived;
- last pushed in 2023;
- uses old `shikimori.me` host in source;
- no TV/device flow;
- no relay/session design;
- no PKCE-specific implementation.

Useful only as confirmation that Shikimori works with standard server-side Authorization Code clients.

License: MIT.

## Search result

No ready-made, maintained GitHub project was found that provides all of:

- Lampa UI;
- Shikimori login;
- secure server-side client-secret handling;
- QR/device pairing;
- polling;
- token refresh and revocation.

The existing Shikimori Lampa plugin improves UX but exposes its OAuth secret publicly. The existing Plex Lampa plugin has the right TV UX but relies on Plex-specific PIN endpoints unavailable in Shikimori.

## Recommended implementation

Combine two patterns without copying the insecure part:

1. Use the Plex plugin's TV interaction model:
   - QR/code overlay;
   - cancel/back;
   - polling status;
   - timeout;
   - manual fallback hidden in advanced settings.
2. Use a ShikiLamp OAuth relay for Shikimori:
   - relay holds client secret;
   - relay creates one-time session and user code;
   - phone follows normal Authorization Code flow;
   - relay validates state and exchanges code;
   - TV consumes result once;
   - refresh goes through relay.
3. Generate QR locally or serve it from our relay. Do not send authorization URLs to public QR-generation services.

## Practical decision

A backendless primary login is rejected. The existing GitHub example proves it can function technically, but only by publishing the secret and requiring manual code entry. That is not acceptable for a public product.

The implementation target remains:

```text
Lampa QR/pairing UI + Cloudflare Worker OAuth relay + Shikimori Authorization Code flow
```

Manual token or OOB-code entry remains a developer/recovery feature only.
